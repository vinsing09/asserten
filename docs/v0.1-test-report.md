## Bugs found and fixed during this autonomous run

1. **agent_kind_inferrer.py prompt template `KeyError: '"kind"'`** — JSON braces in the prompt template conflicted with Python `.format()` placeholders. Fixed by escaping `{{...}}`. Pre-existing bug in agentops-backend, surfaced by asserten's first audit call.

2. **asserten client field mismatch** — sent `raw_system_prompt` but `/agents/draft` expects `system_prompt`. Fixed at the client boundary (translate `raw_system_prompt` → `system_prompt`).

3. **commit_draft response shape** — asserten expected `{id, name}` but backend returns `{agent_id, version_id, agent_name, fixes_applied, ...}`. Updated client to read the real shape.

4. **DEEP optimize query params** — asserten sent `eval_run_id` + `mode` in the JSON body; backend expects them as query params. Fixed.

5. **Test cases require contract first** — surfaced via 400 "Generate contract first" on the first eval attempt. Added `cmd_prepare_eval` that generates contract then test cases in sequence.

6. **CLI stdin block** — `client/cli.py` was reading stdin with `read()` which blocks when invoked from interactive shells without piped input. Fixed with `select.select(timeout=0.05)`.

7. **Single commit creates BOTH v0 and v1** — asserten was originally going to dual-commit (empty + accepted) to manufacture v0 separately. Backend already produces v0 ('original') and v1 ('Audited') from one commit. Removed the dual-commit dance.

8. **Forensics script header text was hardcoded to old sweep tag** — fixed inline (`pipeline_forensics_deep.py` now reads `ARTIFACT_GLOB.name`).

## What's deliberately NOT covered in v0.1

| limitation | impact | next |
|---|---|---|
| `/asserten-failures` is a stub | User sees `failed/total` count from eval summary; no per-case detail. | v0.2 — `/eval-results/{run_id}/failures` endpoint then plugin command. |
| LIGHT optimize requires K candidate v1s | Demos either use `bea9a565`'s historical pool or settle for trivial single-candidate case. | v0.2 — `/asserten-audit-bulk K=10` runs the audit K times in parallel. |
| DEEP optimize requires manual `eval_run_id` | User has to pass it as JSON arg. | v0.2 — auto-resolve from session state. |
| No retry / backoff on flaky judge calls | If a judge errors mid-eval, the eval may report `invalid`. | Backend already handles this via Task A judge-stability gate; surfaced in `EvalSummary.invalid`. |

### What this proves

- **End-to-end DEEP plumbing works through the asserten client.** The `optimize_deep` call submits, the `wait_improvement_job` poll loop blocks correctly, and the job state transitions visible in DB are reflected in the final return value.
- **Tasks A → E hardening (the post-fix stabilization plan) is doing its job.** Both synth patches were caught at verify by the existing replay-majority + verify-gap mechanisms — neither slipped through to produce a regression. This is a "no regression shipped" win, even though the user-facing v2b delta is 0pp.
- **Honest negative result is OK for the demo.** When friend runs `/asserten-optimize-deep` on their agent, the result will be one of:
  - **Patches verified + v2b shows positive delta** → ideal, we have a v2b row to compare.
  - **Patches verified + v2b shows zero/negative delta** → real regression caught by v2 eval (Tasks B.2 / B.1 future work).
  - **0 patches verified** (this case) → tell the user: "Cross-refine couldn't find a non-regressing improvement on this iteration. v2b is unavailable."

### Plugin behavior in the 0-patches case

Right now `cmd_optimize_deep` writes `v2b_version_id=""` when no patches survive verify (because `final.get("result_version_id")` returns nothing). Subsequent `/asserten-eval v2b` will respond "No `v2b` version in session yet." which is correct UX. `/asserten-compare` will show v2b row as `—`.
